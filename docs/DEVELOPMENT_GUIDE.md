### 5. SSL Certificate Setup for Local Development

The application uses HTTPS for local development. You need to trust the self-signed certificates.

#### Certificate Location

Certificates are located in:
```
certificates/
├── server/
│   ├── server.crt    # Server certificate
│   ├── server.key    # Server private key
│   └── server.pfx    # Combined certificate (Windows)
└── qlik/
    ├── client.pem    # Qlik client certificate
    ├── client_key.pem
    └── root.pem      # Qlik root CA
```

---

#### **macOS Setup**

macOS requires certificates to be added to the system Keychain for browsers to trust them.

##### Option A: Using mkcert (Recommended)

[mkcert](https://github.com/FiloSottile/mkcert) automatically creates locally-trusted certificates.

```bash
# Install mkcert via Homebrew
brew install mkcert
brew install nss  # Required for Firefox

# Install the local CA in system trust store
mkcert -install

# Generate certificates for local development
cd certificates/server
mkcert -key-file server.key -cert-file server.crt local.databridge.ch localhost 127.0.0.1 ::1

# Verify installation
mkcert -CAROOT
```

After running `mkcert -install`, your browsers will automatically trust certificates generated by mkcert.

##### Option B: Manual Keychain Installation

If using existing self-signed certificates:

```bash
# Add certificate to System Keychain
sudo security add-trusted-cert -d -r trustRoot -k /Library/Keychains/System.keychain certificates/server/server.crt
```

Or via Keychain Access GUI:

1. Open **Keychain Access** (Spotlight → "Keychain Access")
2. Select **System** keychain in the sidebar
3. Go to **File** → **Import Items**
4. Select `certificates/server/server.crt`
5. Double-click the imported certificate
6. Expand **Trust** section
7. Set **When using this certificate** to **Always Trust**
8. Close and enter your password to confirm

##### Firefox on macOS (Additional Step)

Firefox uses its own certificate store, not the macOS Keychain:

1. Open Firefox → **Preferences** → **Privacy & Security**
2. Scroll to **Certificates** → **View Certificates**
3. Go to **Authorities** tab
4. Click **Import** and select `certificates/server/server.crt`
5. Check **Trust this CA to identify websites**
6. Click **OK**

##### Verify Certificate Trust (macOS)

```bash
# Check if certificate is trusted
security verify-cert -c certificates/server/server.crt

# List trusted certificates
security find-certificate -a -c "local.databridge.ch" /Library/Keychains/System.keychain
```

#### Troubleshooting Certificate Issues

| Problem | Platform | Solution |
|---------|----------|----------|
| "Your connection is not private" | All | Certificate not trusted - follow setup above |
| "NET::ERR_CERT_AUTHORITY_INVALID" | Chrome | Add cert to system trust store |
| "SEC_ERROR_UNKNOWN_ISSUER" | Firefox | Import cert in Firefox settings |
| Certificate works in Chrome but not Safari | macOS | Ensure cert is in **System** keychain, not login |
| "unable to get local issuer certificate" | Node.js | Set `NODE_TLS_REJECT_UNAUTHORIZED=0` (dev only) |
| mkcert certificates not working | All | Run `mkcert -install` again, restart browser |

```bash
curl -k https://local.databridge.ch:7005
```

## Running the Application

### **IMPORTANT: Qlik Session Cookie Prerequisite**

Before running the frontend application, you **MUST** first log in to Qlik Sense to establish a session cookie.

#### Why is this required?

The application uses **cookie-based authentication** with Qlik Sense Enterprise:

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Browser   │────▶│   Frontend  │────▶│   Backend   │────▶│ Qlik Service│
│             │     │   (:7005)   │     │   (:3002)   │     │   (:3001)   │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
       │                                       │                    │
       │  X-Qlik-Session-{vp} cookie          │                    │
       │◀──────────────────────────────────────┤                    │
       │                                       │   Validate session │
       │                                       │───────────────────▶│
       │                                       │                    │
       │                                       │◀───────────────────│
       │                                       │   User info        │
```

1. When you log in to Qlik Hub, Qlik sets a session cookie (e.g., `X-Qlik-Session-localhost`)
2. The frontend sends this cookie to the backend with each request
3. The backend uses this cookie to validate your identity with Qlik via the Qlik Service
4. **Without this cookie, all API requests will fail with "Unauthorized"**

#### Step 1: Log in to Qlik Sense Hub

Open your browser and navigate to:

```
https://qs-internal.databridge.ch/localhost/hub/my/work
```

> **Note**: Replace `localhost` with your virtual proxy name if different.

Log in with your Windows credentials (or the credentials provided by your administrator).

#### Step 2: Verify the Cookie is Set

After logging in successfully:

1. Open browser Developer Tools (F12)
2. Go to **Application** → **Cookies**
3. Look for a cookie named `X-Qlik-Session-localhost`
4. This cookie must be present before running the frontend

#### Step 3: Keep the Browser Session Active

- The cookie has a session timeout (typically 30 minutes of inactivity)
- If you get "Unauthorized" errors, log in to Qlik Hub again
- Some browsers don't share cookies between tabs - use the same browser window

#### Troubleshooting Cookie Issues

| Problem | Solution |
|---------|----------|
| Cookie not set | Log in to Qlik Hub again |
| "Unauthorized" errors | Session expired - re-login to Qlik Hub |
| Cookie domain mismatch | Ensure frontend runs on same domain (local.databridge.ch) |
| Different virtual proxy | Check `startup.json` `vp` matches your Qlik virtual proxy |

## API Development

### Creating a New Endpoint

#### 1. Define Entity (if needed)

Create `src/entities/MyEntity.ts`:
```typescript
export interface MyEntity {
  id: number;
  name: string;
  createdAt: Date;
}
```

#### 2. Create Repository

Create `src/repositories/MyEntityRepository.ts`:
```typescript
import { injectable } from 'tsyringe';
import { DatabaseService } from '../lib/database/DatabaseService';

@injectable()
export class MyEntityRepository {
  constructor(private db: DatabaseService) {}

  async findAll(): Promise<MyEntity[]> {
    return this.db.knex('my_entities').select('*');
  }

  async create(data: Partial<MyEntity>): Promise<MyEntity> {
    const [entity] = await this.db.knex('my_entities')
      .insert(data)
      .returning('*');
    return entity;
  }
}
```

#### 3. Create Service

Create `src/services/MyEntityService.ts`:
```typescript
import { injectable } from 'tsyringe';
import { MyEntityRepository } from '../repositories/MyEntityRepository';

@injectable()
export class MyEntityService {
  constructor(private repository: MyEntityRepository) {}

  async getAll(): Promise<MyEntity[]> {
    return this.repository.findAll();
  }

  async create(data: Partial<MyEntity>): Promise<MyEntity> {
    // Business logic here
    return this.repository.create(data);
  }
}
```

#### 4. Create Controller

Create `src/controllers/MyEntityController.ts`:
```typescript
import { Request, ResponseToolkit } from '@hapi/hapi';
import { controller, httpGet, httpPost } from 'hapi-decorators';
import { injectable } from 'tsyringe';
import Joi from 'joi';
import { MyEntityService } from '../services/MyEntityService';

@controller('/api/my-entities')
@injectable()
export class MyEntityController {
  constructor(private service: MyEntityService) {}

  @httpGet('/')
  async getAll(request: Request, h: ResponseToolkit) {
    const entities = await this.service.getAll();
    return h.response(entities).code(200);
  }

  @httpPost('/')
  async create(request: Request, h: ResponseToolkit) {
    const entity = await this.service.create(request.payload);
    return h.response(entity).code(201);
  }
}
```

#### 5. Register Controller

In `src/index.ts`, ensure the controller is imported (auto-discovery via decorators).

## Code Style & Best Practices

### TypeScript

- Use strict type checking
- Avoid `any` type
- Define interfaces for all data structures
- Use async/await over promises

### Naming Conventions

- **Files**: PascalCase for classes (`UserService.ts`), camelCase for utilities (`stringHelper.ts`)
- **Classes**: PascalCase (`UserService`)
- **Functions**: camelCase (`getUserById`)
- **Constants**: UPPER_SNAKE_CASE (`MAX_RETRIES`)
- **Interfaces**: PascalCase (`IUserRepository` or `UserEntity`)

### Error Handling

```typescript
import Boom from '@hapi/boom';

// Validation error
throw Boom.badRequest('Invalid input');

// Not found
throw Boom.notFound('User not found');

// Unauthorized
throw Boom.unauthorized('Invalid token');

// Server error
throw Boom.internal('Something went wrong', { originalError: err });
```

## Troubleshooting

### Common Issues

#### Port Already in Use

```bash
# Find process using port
lsof -i :3002  # Mac/Linux
netstat -ano | findstr :3002  # Windows

# Kill process or change port in docker-compose.yml
```

#### Database Connection Failed

- Ensure database container is running: `docker-compose ps`
- Check database credentials in `.env`
- Verify database is created: `docker-compose exec db psql -U root -l`

#### Qlik Connection Failed

- Verify Qlik Sense server is accessible
- Check certificate files exist and are valid
- Verify virtual proxy configuration
- Test connection: `curl https://your-qlik-server/hub`

#### Node Modules Issues

```bash
# Remove and reinstall
docker-compose down
docker volume rm db-siemens-dev_db-insight-app-api-node_modules
docker-compose up --build
```

#### TypeScript Compilation Errors

```bash
# Clean build directory
rm -rf sih-atellica-qplus-backend/build
cd sih-atellica-qplus-backend
npm run tsc
```
